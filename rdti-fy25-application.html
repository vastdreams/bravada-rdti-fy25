Core Activity Description (4000 chars)
This core activity covers the brains of Quotech ‚Äì the part that has to understand emails and documents, keep long‚Äëlived context, and serve that context back quickly and reliably for downstream modules (billing, timesheets, BI, etc.). In practical terms, we are building and testing a neuroplasticity‚Äëinspired context engine for construction documents. The system ingests emails and attachments (contracts, quotes, variations, day sheets, invoices, POs, QA records), turns them into structured summaries, stitches them together into a graph, and then uses that graph to classify and retrieve information for real jobs. The R&D work for this core activity includes: - Designing and implementing a context graph and vector memory for thousands of construction documents, with page‚Äëlevel nodes, similarity edges and project relationships. - Investigating whether usage‚Äëdriven edge weighting (our ‚Äúneuroplasticity‚Äù idea) actually improves retrieval over time in the real world. - Building a multi‚Äësignal classifier that uses subject, sender, thread history, body content and attachments together to decide whether an email is a Bill, PO, Contract, Variation, Day Sheet, QA record, etc. - Wiring this context engine into real sub‚Äëmodules: - Billing / PO / Invoice intelligence (matching bills to POs, tagging suppliers and jobs, flagging price anomalies). - Timesheet and day‚Äësheet intelligence (classifying, validating and linking to jobs). - BI and information chat (letting a user ask questions like ‚Äúshow me all variations for Job X‚Äù and getting a grounded answer). The outcome we are aiming for is not just another RAG stack; it is a context system that can live with messy construction data for years, learns which relationships actually matter, and feeds that back into modules that save real time for sub‚Äëcontractors.
Why outcome could not be known in advance
A competent professional in AI/ML and construction software could not have known the outcome of this activity in advance for a few simple reasons: 1. Nobody has tried this specific combination in this domain. There are plenty of papers and products about RAG, vector search and ‚ÄúAI for documents‚Äù, but we could not find any prior work that combines graph + vector + usage‚Äëbased edge updates in a live construction environment and targets the very specific document mix that sub‚Äëcontractors deal with. 2. Classification behaviour for construction categories is unknown. A professional could reasonably guess that a model will be ‚Äúokay‚Äù for Bills and POs, but has no way to know upfront what accuracy you can reach for things like differentiating Day Sheets from Timesheets when people change templates and naming, or teasing apart QA records from general emails. Those are empirical questions. 3. Latency vs. depth trade‚Äëoffs can‚Äôt be solved on paper. You can sketch architectures all day, but you can‚Äôt know in advance if this particular stack (summarisation ‚Üí embedding ‚Üí graph traversal ‚Üí LLM) will stay under 500 ms P95 on real workloads and still keep enough context to answer real questions reliably. That depends on real data volume, query patterns, hardware and implementation details. 4. ‚ÄúNeuroplastic‚Äù memory is still a hypothesis. The idea of strengthening edges based on use is reasonable, but nobody could say beforehand whether it would measurably improve retrieval accuracy or just overfit to noisy usage patterns. That behaviour only emerges after weeks of real queries. 5. Cross‚Äëcompany transfer is genuinely unknown. It was not clear before we started how much accuracy would drop if you trained on Company A‚Äôs emails and then applied the same system to Company B, or how many documents would be needed to fine‚Äëtune back to an acceptable level. Because of these uncertainties, the only way to answer ‚Äúwill this actually work in construction, at the accuracy and speed we need?‚Äù was to design experiments, run them, and look at the data.
Hypothesis
The working hypothesis for this core activity is: If we combine a graph‚Äëbased memory of construction documents with vector embeddings and usage‚Äëdriven edge weighting, and run classification and retrieval on top of that, then we can (a) classify core construction documents with more than 85% accuracy and (b) answer context‚Äëheavy questions with sub‚Äësecond latency, and (c) see those metrics improve as the system is used. Under that umbrella we are testing several concrete propositions: - That decomposing documents into page‚Äëlevel nodes with summaries and explicit relationships will give better context for LLM answers than a flat vector index alone. - That a multi‚Äësignal classifier (subject + sender + thread + content + attachments) will materially outperform any single‚Äësignal classifier for Bills, POs, Contracts, Variations, Day Sheets and QA records. - That a three‚Äëtier confidence routing (auto‚Äëassign, suggest, manual) can hit a useful balance between automation and safety for sub‚Äëcontractors. - That edge weighting based on retrieval usage will, over about a month of real‚Äëworld use, lead to higher retrieval accuracy and shorter average paths through the graph.
Experiment design
We ran a sequence of experiments, each aimed at one part of the hypothesis: 1. Summarisation model tests. We took 500 real documents across categories and ran them through different summarisation models (T5, Janus, GPT). Humans rated faithfulness and usefulness; we compared this to runtime. This told us which models were ‚Äúgood enough‚Äù to sit at the front of the pipeline. 2. Embedding and clustering. We embedded 2,000 page‚Äëlevel summaries and looked at how well simple clustering aligned with our manual labels (Bills, Contracts, etc.). This checked that the embedding space even had the structure we needed. 3. Soft‚Äëedge creation and GPT refinement. We used cosine similarity to propose ‚Äúsimilar page‚Äù pairs, then sent batches to GPT to label as strong or weak relationships, with short reasons. We compared those labels to human judgements and tuned thresholds. 4. Classification experiments. On 1,500 manually labelled emails/documents we tried single‚Äësignal and multi‚Äësignal classifiers and measured precision/recall/F1 per category. This told us how much multi‚Äësignal actually buys us. 5. Retrieval and latency tests. Using 500 ‚Äúknown answer‚Äù queries from pilot companies, we measured answer accuracy and latency at P50/P95/P99, before and after adding parallel graph traversal and Redis caching. 6. Neuroplasticity / usage‚Äëdriven evolution. We deployed the system at a pilot company and let it run for four weeks. Every week we sampled queries, checked answer quality, and logged which edges were used. We then updated edge weights and watched how accuracy and path lengths changed over time.
Evaluation
We evaluated the experiments using hard numbers and practical ‚Äúdoes this feel right in the field‚Äù checks. On the quantitative side we tracked: - Classification accuracy by category and overall, compared against explicit targets. - Retrieval latency at P50, P95 and P99, and how that changed with parallelism and caching. - Relationship precision before and after GPT refinement. - ‚ÄúLeakage‚Äù rate, i.e. how often human users had to correct the system‚Äôs decisions. - Memory evolution, i.e. improvement in accuracy and reduction in path length over weeks. On the qualitative side we: - Sat with operations staff at pilot companies and asked them to mark AI classifications as right/wrong/close. - Reviewed tricky retrieval cases (for example ‚Äúfind all variations linked to this PO‚Äù) and asked whether the context the system pulled in looked like what a competent project manager would have pulled manually. - Used the chain‚Äëof‚Äëthought logs to understand why the system took certain paths through the graph, and whether those paths made sense to humans. We did not use formal p‚Äëvalues; the decision standard was more practical: is the system now reliable enough that a sub‚Äëcontractor would trust it for real work, and are we clearly above what they could do with their existing tools?
Conclusions
By the end of FY25 we had enough data to draw some clear technical conclusions: 1. The neuroplasticity idea works in practice. When we ran the system for a month at a pilot company and let edges strengthen based on use, retrieval accuracy improved by around 7‚Äì9 percentage points and average path length dropped. Irrelevant results went down materially. 2. Multi‚Äësignal classification is worth the complexity. Bills, POs and Contracts passed the 90% accuracy mark; lower‚Äëfrequency categories lagged but still improved. A single‚Äësignal baseline simply did not get there for construction data. 3. We had to pivot away from a ‚Äúpure‚Äù memory tree. An early design that tried to store everything in a generic memory tree did not hit latency or maintainability targets. Moving to a hybrid of SQL for structured facts plus a graph for relationships was a key architectural pivot. 4. Parallelisation and caching were non‚Äëoptional. Without them we could not meet P95 latency targets. This will shape all future scaling work. 5. Handwritten documents remain a hard problem. For handwritten day sheets and certain scanned content the system still underperforms. That is clearly marked as out of scope for FY25 and a topic for later research. Overall, the core conclusion is that the context‚Äëgraph + multi‚Äësignal classifier + neuroplastic memory architecture is viable for construction email/document intelligence, and we now know exactly where it is strong and where it still needs work.
New knowledge
The main pieces of new knowledge from this core activity are: 1. How to build a usage‚Äëevolving memory for construction documents. We now have concrete patterns for structuring document and page nodes, creating and refining similarity edges, and updating edge weights over time in a way that measurably improves retrieval. 2. Signal weighting and thresholds for construction email classification. We learned which signals matter most, how to weight them, and what confidence thresholds are realistic for different document categories. 3. A practical hybrid of SQL and graph for this domain. We have working schema patterns and routing rules for ‚Äúwhat lives in SQL vs what lives in the graph‚Äù, which can be reused in other sub‚Äëcontractor products. 4. Concrete limits of transfer learning between sub‚Äëcontractors. We quantified how much accuracy is lost when you move a model between companies, and how many documents you need to claw that back. These are written up in our internal research notes and sit behind the product features we are shipping.

Core Activity Description
This core activity focuses on the front door of cost intelligence: reading quotes from suppliers, no matter how they are formatted, and turning them into clean, structured cost data that can drive billing, margin analysis and anomaly detection. In practice, we see quotes arriving as: - Excel spreadsheets with different column layouts and naming conventions; - PDFs exported from those spreadsheets with different fonts and formatting; - Scans of typed or annotated documents; and - The occasional image or email‚Äëembedded table. The R&D problem is to work out whether we can: - recognise these different formats; - extract all the important fields (item, description, quantity, unit, unit price, total, category, etc.) with high accuracy; and - then build reliable analytics on top of that, such as margin forecasts and price anomaly flags. The work under this core activity includes: - Analysing real quotes from pilot companies and building a taxonomy of formats and column names. - Designing an OCR + LLM pipeline that can handle both native digital quotes and scanned documents. - Developing fuzzy matching for column headers so that ‚ÄúQty‚Äù, ‚ÄúQuantity‚Äù, ‚ÄúUnits‚Äù, ‚ÄúNo.‚Äù, etc. all land in the right canonical field. - Training classifiers to assign each line item to a cost centre (Labour, Materials, Equipment, Subcontract, Other, with construction‚Äëspecific sub‚Äëcategories). - Building prototype predictive models that compare quotes to actuals and highlight where a job is likely to run over. - Developing pricing anomaly detection that flags unusual unit rates or total amounts relative to historical behaviour by supplier.
Hypothesis
The working hypothesis for this core activity is: If we combine OCR, LLM‚Äëbased parsing, fuzzy column mapping and construction‚Äëspecific classifiers, then we can turn messy quotes into structured cost data with high enough accuracy to support reliable billing, BI and anomaly detection. More concretely, we are testing whether: - We can reach around 96% field‚Äëlevel accuracy on structured Excel‚Äëorigin quotes, and stay above 80% on typed PDFs. - We can reliably map dozens of real‚Äëworld column header variants into a small canonical schema. - We can classify line items into cost centres with more than 85% accuracy overall. - We can flag unusual prices with a high detection rate and an acceptable false positive rate (targeting more than 90% detection and less than 5% false positives).
